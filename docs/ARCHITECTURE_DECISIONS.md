# Архітектурні Рішення

## 1. Призначення документа

Цей документ пояснює не лише "як влаштований код", а й "чому він влаштований саме так". Він фіксує ключові архітектурні рішення, їх причини, переваги, компроміси та можливі альтернативи.

## 2. ADR-1: Багаторівнева архітектура замість монолітного `main.py`

### Рішення

Застосунок розділено на шари:

- `web`
- `services`
- `repositories`
- `infrastructure`

### Чому це зроблено

Початкова консольна або "плоска" архітектура швидко стає складною для підтримки, коли з'являються:

- кілька сутностей
- форми з валідацією
- окремі сценарії редагування і видалення
- тести
- веб-інтерфейс

### Переваги

- читабельніша структура
- краща тестованість
- менше змішаних відповідальностей
- простіше масштабувати

### Компроміси

- більше файлів
- вищий поріг входу для новачка
- більше "службового" коду

## 3. ADR-2: Flask + Jinja замість SPA-фреймворку

### Рішення

Вибрано серверний рендеринг через Flask і Jinja, а не React/Vue SPA.

### Причини

- проєкт навчальний і CRUD-орієнтований
- основний акцент на backend-логіці, БД і архітектурі
- серверний HTML простіше відлагоджувати
- менша залежність від складного frontend-build pipeline

### Виграші

- швидший старт розробки
- простіший deployment
- зручніші PRG-сценарії
- менше стану на клієнті

### Обмеження

- менше інтерактивності, ніж у повноцінній SPA
- частина UI-state тримається через query params і session payload

## 4. ADR-3: Прямий SQL через `PyMySQL` замість ORM

### Рішення

Робота з MySQL реалізована через прямі SQL-запити.

### Причини

- прозорість SQL для навчального проєкту
- легкість пояснення схеми БД
- менше прихованої магії, ніж в ORM

### Переваги

- повний контроль над запитами
- зрозумілий зв'язок між таблицею і кодом
- простіше пояснювати на захисті

### Недоліки

- більше ручного мапінгу
- більше шансів на дублювання
- складніше масштабувати в дуже великому проєкті

## 5. ADR-4: `services` як окремий шар бізнес-логіки

### Рішення

Складні правила не зберігаються у controller або repository.

### Що саме винесено в сервісний шар

- обчислення `total_price`
- розрахунок `due_amount`
- нормалізація статусів
- складання preview і receipt-даних
- централізація CRUD-правил

### Чому це добре

- controller стає тонким
- unit-тести пишуться простіше
- бізнес-правила не прив'язані до HTTP

## 6. ADR-5: PRG для всіх write-сценаріїв

### Рішення

Після `POST` застосовується `redirect`.

### Причина

Без цього користувач може:

- дублювати записи при `F5`
- випадково повторювати форму
- потрапляти в нестабільний стан UI

### Переваги

- стійкі CRUD-сценарії
- кращий UX
- менше регресій після submit

## 7. ADR-6: Modal-first UX для CRUD

### Рішення

Основні create/update сценарії реалізовані через modal-вікна, а не окремі сторінки.

### Причини

- швидший робочий ритм
- збереження контексту таблиці
- компактніший UX

### Компроміси

- складніший UI-state
- більше умов для повторного відкриття modal після помилки
- сильніша залежність між query params, шаблоном і JavaScript

## 8. ADR-7: Стабільні статусні коди в БД

### Рішення

Для замовлень на рівні БД використовуються технічні коди, а не локалізований текст.

### Приклад

- `paid` / `unpaid`
- `ready` / `not_ready`

### Причини

- менше ризику проблем кодування
- легше змінювати мову UI
- простіше тестувати й нормалізувати дані

## 9. ADR-8: Модульний vanilla JavaScript замість одного великого файлу

### Рішення

Frontend-логіка розбита на кілька JS-модулів.

### Причини

- краще підтримувати
- простіше локалізувати баги
- менше ризику зламати іншу частину інтерфейсу

### Ключові модулі

- `modals.js`
- `orders.js`
- `previews.js`
- `shell.js`
- `tables.js`
- `toasts.js`
- `utils.js`

## 10. ADR-9: Кастомні error pages як частина продуктового UX

### Рішення

Помилки `404` і `500` мають власний branded-дизайн.

### Причини

- вимога до інтерфейсу
- цілісність візуального стилю
- демонстрація продуманого UX навіть у помилкових сценаріях

## 11. ADR-10: Документація як окремий артефакт проєкту

### Рішення

Документація не обмежується одним README, а винесена у кілька спеціалізованих файлів.

### Причини

- різні аудиторії потребують різного рівня деталізації
- технічний опис не повинен змішуватись із user guide
- база даних і тестування потребують окремих довідників

## 12. Альтернативи, які могли б бути обрані

### FastAPI

Підійшов би для більш API-центричної системи, але тут немає потреби в сильному акценті на REST/JSON.

### Django

Дає багато готового, але для цього проєкту був би важчим за потребу.

### SQLAlchemy

Корисний для великого застосунку, але для навчального CRUD може приховати важливу SQL-логіку.

### React SPA

Дав би багатший frontend, але різко збільшив би складність стеку.

## 13. Поточні борги, що випливають з архітектурних рішень

- частина текстів ще потребує повного очищення від наслідків проблем кодування
- modal-state досить щільно пов'язаний з query parameters
- немає окремого migration framework
- відсутня production-grade auth-модель

## 14. Коли варто переглядати ці рішення

Архітектуру доцільно переглянути, якщо:

- з'явиться багатокористувацький режим
- додасться повноцінна авторизація
- потрібно буде будувати публічний API
- обсяг логіки компонентів суттєво зросте
- з'явиться потреба в масштабованому продакшн-розгортанні
