# Безпека І Валідація

## 1. Призначення документа

Документ описує, як у проєкті реалізовані:

- безпечне зберігання конфігурації
- валідація вводу
- базові механізми захисту від помилкових даних
- відомі обмеження в безпеці

## 2. Конфігурація бази даних

### Поточний підхід

Паролі до БД не зберігаються в коді. Вони читаються через:

- змінні середовища
- файл `.env`

### Чому це важливо

Hardcoded пароль:

- легко випадково закомітити
- важко змінювати між середовищами
- погано масштабується

### Які змінні використовуються

- `DB_HOST`
- `DB_USER`
- `DB_PASSWORD`
- `DB_NAME`
- `APP_SECRET_KEY`

## 3. Валідація на клієнті

### Де реалізована

У HTML-формах через:

- `type`
- `pattern`
- `required`
- `autocomplete`
- `inputmode`
- `title`

### Що це дає

- ранній зворотний зв'язок користувачу
- менше невалідних submit
- кращий UX

### Обмеження

Клієнтську валідацію не можна вважати достатньою, бо її можна обійти ручним POST-запитом.

## 4. Валідація на сервері

### Чому вона обов'язкова

Сервер є джерелом істини. Навіть якщо браузер пропустив або обійшли клієнтську перевірку, сервер повинен сам перевірити дані.

### Що зараз перевіряється

#### Клієнти

- формат телефону
- формат email
- наявність обов'язкових полів
- коректність дати народження

#### Збірки

- наявність усіх потрібних компонентів
- коректність ідентифікаторів компонентів
- коректність типу збірки

#### Замовлення

- наявність клієнта
- наявність збірки
- коректність дати завершення
- допустимий часовий інтервал
- коректність статусів

#### Комплектуючі

- обов'язковість ключових полів
- правильні числові типи
- базова коректність значень

## 5. Regex-правила

### Телефон

Очікуваний формат:

```text
+380XXXXXXXXX
```

Приклади валідних значень:

- `+380501234567`
- `+380671112233`

Приклади невалідних значень:

- `0501234567`
- `380501234567`
- `+38abc`

### Email

Очікується коректний email із доменною частиною, наприклад:

- `user@example.com`
- `admin@test.org`

Некоректні приклади:

- `test@example`
- `name@`
- `plain-text`

## 6. UX при помилці валідації

Коли користувач помиляється, система має поводитись м'яко:

- modal не закривається
- поля не очищаються
- помилка показується inline
- користувач бачить, яке саме поле треба виправити

Це зменшує фрустрацію й відповідає вимозі дружнього UX.

## 7. Обробка системних помилок

### 404

Використовується кастомна branded-сторінка для неіснуючих URL.

### 500

Використовується окрема branded-сторінка для внутрішніх винятків.

### Чому це корисно

- користувач не бачить "сирий" stack trace
- інтерфейс лишається цілісним
- демонструється продуманий підхід до обробки збоїв

## 8. Що вже захищено на рівні архітектури

- пароль БД винесено з коду
- CRUD-форми валідовані на сервері
- після POST використовується redirect
- статуси замовлень нормалізовані
- критичні поля мають перевірку форматів

## 9. Що ще не реалізовано

Поточна безпека є базовою, а не production-grade.

Поки відсутні:

- повноцінна авторизація
- ролі користувачів
- CSRF-захист
- rate limiting
- аудит дій
- централізований secrets manager

## 10. Основні ризики

| Ризик | Поточний статус |
|---|---|
| hardcoded пароль | усунуто |
| невалідний email | блокується |
| невалідний телефон | блокується |
| дублювання POST при F5 | усунуто через PRG |
| ручний ввід ID замість вибору | замінено dropdown |
| витік stack trace користувачу | зменшено через custom 500 |

## 11. Що рекомендується для наступного етапу

1. Додати CSRF-захист.
2. Додати автентифікацію.
3. Розділити ролі `admin` і `operator`.
4. Додати журнал дій.
5. Впровадити ширші обмеження БД: `UNIQUE`, суворі типи, додаткові `CHECK`.

## 12. Висновок

Для навчального CRUD-проєкту поточний рівень валідації та безпеки є хорошим базовим рівнем. Для реального продакшн-використання систему потрібно буде посилювати окремим безпековим етапом.
