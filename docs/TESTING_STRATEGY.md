# Стратегія Тестування

## 1. Призначення документа

Цей документ описує, як у проєкті організовано тестування, що вже покрито, які сценарії вважаються критичними і що ще варто перевіряти в майбутньому.

## 2. Цілі тестування

Тестування в цьому проєкті має підтверджувати:

- коректність CRUD-операцій
- стабільність маршрутизації
- відсутність регресій у валідації форм
- коректний `POST -> Redirect -> GET`
- правильну роботу сервісного шару
- базову цілісність взаємодії `controller -> service -> repository`

## 3. Поточний стек тестування

- `pytest`
- Flask `test_client`
- fake/stub залежності для частини сценаріїв

## 4. Рівні тестів

### 4.1. Unit tests

Перевіряють:

- ізольовану бізнес-логіку сервісів
- розрахунок цін
- формування сум до сплати
- нормалізацію статусів
- допоміжні правила валідації

### 4.2. Integration tests

Перевіряють:

- реальні HTTP-маршрути
- Flask routing
- рендер сторінок
- поведінку форм і редіректів
- відображення modal-state після помилок

### 4.3. Manual smoke tests

Рекомендуються для перевірки:

- модалок
- JavaScript preview
- адаптивності
- реального UX у браузері
- custom error pages

## 5. Критичні бізнес-сценарії

Нижче наведено сценарії, які мають першочергову цінність.

### 5.1. Клієнти

- створення валідного клієнта
- блокування невалідного телефону
- блокування невалідного email
- збереження введених значень при помилці
- редагування існуючого клієнта
- видалення клієнта

### 5.2. Збірки

- створення збірки з усіма потрібними компонентами
- обчислення `total_price`
- редагування збірки
- видалення збірки

### 5.3. Замовлення

- створення замовлення для існуючого клієнта
- обчислення `production_time`
- обчислення `due_amount`
- редірект після успішного створення
- показ чека після операції
- редагування замовлення
- видалення замовлення

### 5.4. Помилки

- `404` повертає кастомну сторінку
- `500` повертає кастомну сторінку

## 6. Що вже підтверджено тестами

На момент останньої перевірки:

```text
23 passed
```

Покрито:

- головна сторінка
- сторінки клієнтів, збірок, замовлень
- створення клієнта
- створення збірки
- створення замовлення
- inline-помилки форм
- regex-валідація email і телефону
- custom error handlers
- частина CRUD-сценаріїв

## 7. Матриця перевірок

| Область | Тип тесту | Поточний стан |
|---|---|---|
| Client form validation | integration | є |
| Build creation | integration | є |
| Order creation | integration | є |
| Service calculation rules | unit | є |
| Error handlers | integration | є |
| Browser JS interactions | manual | частково |
| Responsive layout | manual | частково |
| Docker startup | manual/smoke | бажано |
| Real MySQL integration | частково | бажано розширити |

## 8. Чого поки не вистачає

Поточний набір тестів ще не закриває все.

Не вистачає:

- e2e браузерних тестів
- перевірки адаптивності на кількох розмірах екрана
- тестів на реальну production-подібну БД
- перевірки SQL-тригерів у live-середовищі
- автоматичної перевірки accessibility

## 9. Рекомендований ручний smoke-check перед демонстрацією

Перед показом системи варто пройти короткий чек-лист:

1. Відкрити `/`
2. Відкрити `/clients` і створити клієнта
3. Відкрити `/builds` і створити збірку
4. Відкрити `/orders` і створити замовлення
5. Перевірити редагування всіх трьох сутностей
6. Перевірити видалення
7. Відкрити `/debug/error/404`
8. Відкрити `/debug/error/500`
9. Перевірити кілька мобільних розмірів екрана

## 10. Приклади корисних додаткових тестів у майбутньому

### 10.1. Негативні сценарії

- замовлення з неіснуючим `client_id`
- замовлення з неіснуючим `pc_build_id`
- збірка з неіснуючим компонентом
- редагування неіснуючого запису

### 10.2. Дані на межі

- дуже довге ім'я клієнта
- крайні допустимі дати завершення замовлення
- нульова або аномальна ціна компонента

### 10.3. Інфраструктурні сценарії

- недоступна БД при старті
- недоступна БД під час запиту
- некоректна конфігурація `.env`

## 11. Порядок дій при регресії

Якщо після зміни коду щось ламається, рекомендований порядок такий:

1. запустити `py -m pytest -q`
2. локалізувати, який шар зламаний: `web`, `service`, `repository`, `template`, `js`
3. перевірити останні змінені файли
4. відтворити сценарій вручну
5. додати тест, який ловить саме цю регресію
6. виправити код
7. повторно прогнати тести

## 12. Висновок

Поточне тестування вже достатнє для навчального і демонстраційного рівня проєкту, але для вищого рівня надійності бажано додати e2e, browser automation і ширше покриття негативних сценаріїв.
