# Звіт інспектора / тестувальника (Code Review & QA Report)

## 1. Керування даними (CRUD Операції)
*   **Проблема:** Наразі система дозволяє лише створювати сутності.
*   **Завдання:** Необхідно реалізувати повний цикл **CRUD** (Create, Read, Update, Delete) для всіх ключових елементів системи:
    *   Користувачі (*Users*)
    *   Замовлення (*Orders*)
    *   Збірки (*Builds*)
    *   Комплектуючі (*Components*)

---

## 2. Обробка часу та валідація даних
*   **Проблема:** При створенні замовлення час роботи задається у вигляді числового значення. Введення занадто великого числа (наприклад, `12562237889876`) призводить до падіння сервера з помилкою **500 Internal Server Error** (переповнення типу даних / відсутність валідації).
*   **Рішення:** 
    *   **Мінімальна вимога:** Додати валідацію вхідних даних та обробку винятків, щоб уникнути помилки 500.
    *   **Рекомендоване рішення:** Змінити формат введення часу роботи з числових значень на повноцінний вибір дати та часу (компонент календаря на UI).

---

## 3. Обробка помилок (Error Handling)
*   **Проблема:** При виникненні помилок типу `404 Not Found` або `500 Internal Server Error` користувачеві відображається стандартна дефолтна сторінка `/error`.
*   **Завдання:** Розробити кастомні, інформативні та візуально привабливі сторінки помилок для покращення користувацького досвіду (UX).

---

## 4. Валідація полів введення (Regex Фільтри)
*   **Проблема:** Відсутня належна валідація контактних даних на клієнтській та серверній сторонах.
    *   **Телефон:** Поле дозволяє вводити довільний текст. Необхідно впровадити маску або Regex-валідator для суворого формату введення.
    *   **Email:** Поле пропускає некоректні адреси (наприклад, `jk@jk` без доменної крапки). 
*   **Завдання:** Налаштувати перевірку полів за допомогою регулярних виразів (REGEX) для забезпечення цілісності даних.

---

## 5. Оптимізація користувацького інтерфейсу (UI/UX)

### 5.1. Розділення полів ПІБ
*   **Проблема:** У формі створення клієнта використовується одне поле "ПІБ", хоча початкові клієнти мають лише Ім'я та Прізвище. Це викликає плутанину.
*   **Рішення:** Замінити одне поле "ПІБ" на дві окремі колонки: **Прізвище** та **Ім'я**.

### 5.2. Відображення помилок валідації форми
*   **Проблема:** Якщо користувач припускається помилки (наприклад, вводить email, який вже існує, або перевищує ліміт символів), форма закривається, а повідомлення про помилку з'являється десь збоку як коротке сповіщення (toast), яке швидко зникає. Це незручно.
*   **Рішення:** Не закривати форму у разі помилки. Виводити повідомлення про помилку безпосередньо над самою формою або біля відповідного поля, щоб користувач міг одразу її виправити.

---

## 6. Поради щодо рефакторингу (Refactoring)

*   **а) Декомпозиція `app.js`:**  
    Файл `app.js` став занадто великим і перевантаженим логікою. Його необхідно розділити на кілька менших, вузькоспеціалізованих класів або модулів.
*   **б) Оптимізація `app.py` та архітектура:**  
    *   Виклики методів `template.renderer()` містять занадто довгі списки параметрів, що засмічує код. Рекомендується створити окремий клас (наприклад, `CustomRenderer`), який приховає внутрішню рутину, дозволяючи викликати лаконічний метод:  
        `customrenderer.render(object, ..., "home.html")`
    *   Розділити `app.py` за архітектурним шаблоном MVC, винісши логіку обробки запитів в окрему директорію `controllers`.
*   **в) Рефакторинг сервісного шару:**  
    За можливості провести декомпозицію занадто довгих методів у сервісах, розбивши їх на менші приватні підфункції з єдиною відповідальністю (Single Responsibility Principle).

---

## 7. Додаткові рекомендації (DevOps & Testing)

*   **а) Тестування:** Написати базове покриття проєкту юніт-тестами (Unit tests) для бізнес-логіки та інтеграційними тестами (Integration tests) для перевірки взаємодії компонентів.
*   **б) Контейнеризація:** Створити конфігураційний файл `docker-compose.yaml` для швидкого, зручного та ізольованого запуску всього проєкту (разом із базою даних) на будь-якому середовищі.
